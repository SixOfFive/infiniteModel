# InfiniteModel changelog

Moved here from the `VERSION =` lines in `server.py` / `client.py` to keep the source small. Newest first. When bumping a version, add the entry at the top of the relevant section and keep `VERSION` in the source as just the `0.2-mXYZ` tag.

## client.py (0.2-m4c34)

m4c34 **#public-release: config-driven hosts + token from a gitignored file (no secrets baked in).** Worker side of the central-config refactor: imports `load_config`/`gitlab_token`/`gitlab_file_api` from wire; the self-update `_fetch_repo_file` now builds the GitLab URL from `config.json` and reads the token from `$GITLAB_TOKEN` / gitignored `gitlab_token.txt` (the hardcoded `GITLAB_TOKEN`/`GITLAB_FILE_API` constants are deleted). `--controller`/`--control-port` argparse defaults come from `config.json`; `config.json` added to EXTRA_UPDATE_FILES so it syncs like a module. Launchers (`client.sh`/`.bat`, `install/start-client.*`) drop the baked `--controller <IP>`.

## client.py (0.2-m4c33)

m4c33 **#moe-cache: move `_fuse_moe_experts` to the shared `wire.py`.** No behavior change for the worker — the function is now imported from `wire` (added to the `from wire import …` line; the duplicate def removed). The point: the controller shard-cache compile (shards m4c95) now calls the SAME `_fuse_moe_experts` to fuse a per-expert MoE checkpoint into the model's 3D layout, so a non-fused MoE cache (Mixtral/OLMoE) is bit-identical to a cold load by construction. Worker self-update fetches `client.py`+`wire.py` atomically, and the new `wire.py` is a strict superset, so the move is deploy-safe. Validated: py_compile.

## client.py (0.2-m4c32)

m4c32 **#moe-cache Phase B (serve-from-cache): worker installs pre-packed int4 layers directly.** When the controller flags a load `cache=int4` (a VERIFIED int4 shard cache exists, server m4c93), the worker fetches PRE-PACKED units (`/weights?cache=int4`) and builds the resident holders WITHOUT the bf16 stream or any per-layer quant/fuse: new `_install_cached` maps each `<lin>.weight.{qweight,scale,zero}` (2D) → `QuantLinear4` in place of the meta `nn.Linear` and each fused `<experts>.{gate_up,down}_proj.{qweight,scale,zero}` (3D) → `Packed4Tensor3D`, reading `in_features` from the meta module it replaces (no manifest needed worker-side); bf16 passthrough (norms/biases/embed/head) installs via `load_state_dict(assign)` + `_assign_meta_from_sd`; sets `_experts_implementation="eager"` on each touched MoE block. The streamed-expert paths (`stream_experts`/`_nf`) and `_quant_after` are force-skipped in cache mode (`_do_install` dispatcher); the post-loop meta-guard catches any unmaterialized tensor. Whole-layer cache only; pipeline + int4 only (the controller never flags TP — its slices are non-contiguous). Bit-identical by construction (the cache is `pack_linear_int4`/`_pack4_3d` output, proven == load-time quant). Skips the ~4x bf16 stream + re-quant — the big-MoE load win. Validated: py_compile.

## client.py (0.2-m4c28)

m4c28 **#moe-offload (MVP): intra-layer MoE placement — attention on GPU, routed experts in CPU RAM.** The llama.cpp `--override-tensor experts=CPU` strategy, intra-layer. Placement was whole-layer (`_place_modules`: a spilled MoE layer dragged its small, hot, every-token attention to CPU WITH its huge experts). Now, when the controller sends `moe_offload` (pipeline path only), a MoE layer that won't fit GPU whole is SPLIT: attention + norms → GPU, the MoE block (router + routed experts + shared) left on CPU. New worker pieces: `_find_moe_block(layer)` (the direct child holding int4 `Packed4Tensor3D` experts — gated to int4 since those are always heap, so unload-reclaim is unaffected; bf16/mmap experts fall back to whole-layer); `_MoEDeviceBridge` wraps the block so the layer forward moves hidden GPU→CPU around it and the result back (decode hidden is a few KB → negligible); `_place_modules` hybrid branch charges only the MIXER (whole layer − MoE block) + KV to the GPU budget for a split layer, moves only the non-MoE children to GPU (no transient whole-layer GPU spike), installs the bridge; `_mod_gpu_bytes` (device-accurate) replaces `_mod_bytes` for `gpu_bytes` so a split layer reports only its GPU mixer (keeps #95 coexistence honest); a loud post-split assertion fails the load if any expert buffer landed on GPU. **Honest payoff: a VRAM-FIT / coexistence win (ctx-gated by KV) + long-context prefill; decode is ~neutral (the active-expert GEMM stays on CPU).** Pipeline + int4 only; TP path ignores the flag. Off by default (controller opt-in). Validated: py_compile + live on a dense MoE.

## client.py (0.2-m4c27)

m4c27 **#95 coexistence: the worker honors the controller's committed-aware GPU budget (no more grabbing a co-resident model's VRAM).** `_place_modules` gained a `gpu_budget_gb` param (threaded from the load message via `_build_shard` → `from_stream`). When ≥0 it CLAMPS the visible free VRAM (`free = min(free, budget·GB)`) right after `mem_get_info`, so EVERY downstream GPU decision keys off it — the `mode=auto` whole-on-GPU check (`free*0.85`), the hybrid default budget, `live_free`, and the placement string. The bug: a co-resident model's card LOOKS free until it faults its full-ctx KV (allocated lazily at decode), so a 2nd shard's `_place_modules` (which sized against LIVE free VRAM) consolidated onto that VRAM and OOM'd the resident model's decode (qwen3 + 14b). Now budget 0 (GPU fully committed elsewhere) → free 0 → all layers spill to CPU (no grab); a partial budget caps GPU placement to exactly the planner's per-node `free_vram_after_resident − floor`. `<0` = no value sent (old controller) → uncapped, unchanged behavior. Threaded into both the pipeline and TP `from_stream` calls (the controller currently sends it on the pipeline path; the TP path keeps its own per-rank `device=cpu` handling). Paired with server m4c73.

## client.py (0.2-m4c25)

m4c25 **No more /dev/shm temp files: workers stream each weight slice straight into RAM bytes -> heap tensors.** The per-layer streaming loader (#59) staged each slice in a `/dev/shm` tmpfs file and mmap-loaded it (the "zero-copy ~1x transient" rationale). But tmpfs IS RAM, so this (a) double-charged memory the instant a slice landed, (b) capped a node at its (smaller) `/dev/shm` size INDEPENDENT of free RAM, ENOSPCing mid-load — the `FileNotFoundError: /dev/shm/im_w_*.st` build failure seen on theocomp (16 GB shm vs 31 GB RAM), and (c) for CPU-resident layers cloned the mmap to heap anyway (`_drop_slice_mmap`), so there was no 1x win to preserve. Now `fetch()` (both the pipeline and the TP `/weights_tp` path) always returns `_http_get(url)` bytes; `_install` `st_load`s them into HEAP tensors and frees the buffer per layer (~2x ONE layer's bf16 transient — identical to the long-proven Windows/no-shm path). `_drop_slice_mmap` is now a NO-OP (it was a redundant full-layer clone that DOUBLED per-layer transient once weights arrive as heap). `_bound_K()` clamps the prefetch window by FREE RAM only (no separate, smaller tmpfs fs cap). Net: workers use their FULL RAM (not the ~16 GB shm ceiling), there are no temp files on workers at all, and the temp-file / ENOSPC / cleanup-race failure mode is eliminated. (`_http_get_to_file` is now unused; kept defined. The tiny `im_cfg_` config dir — config.json + any remote .py on /tmp disk, not /dev/shm — is unchanged; it is required by `AutoConfig.from_pretrained` and removed immediately.)

## server.py (0.2-m4c88)

m4c88 **Shard compile now supports nvfp4-source checkpoints (e.g. Qwen3.6-27B-NVFP4).** `compile_shards` rejected compressed-tensors NVFP4 (`ValueError: nvfp4-source shard compile not yet supported`), so `qwen3.6-27b-nvfp4` couldn't be pre-compiled. Two pieces beyond `_get_bf16`: (1) `_out_name()` renames `…weight_packed` → `…weight` so the renamed Linear matches `is_layer_lin` and gets quantized like every other layer Linear; (2) the `…weight_scale`/`…weight_global_scale` sidecars are already auto-skipped by `_is_fp8_meta_name` (matches any trailing `*scale*`). embed/norm/lm_head stay bf16 (not packed in this checkpoint family — same assumption the serve path makes). MoE-source compile still raises (per-expert serve-dequant is the remaining follow-up); TP-mode nvfp4 serving is unchanged (separate path, still pipeline-only). Validated: py_compile + live compile of qwen3.6-27b-nvfp4.

## server.py (0.2-m4c86)

m4c86 **Fix "Task was destroyed but it is pending!" — keep refs to resilient-serve bridge tasks.** `_resilient_serve._accept_loop` did `asyncio.create_task(_bridge(conn))` per accepted connection WITHOUT keeping a reference. asyncio holds only a weak ref to a bare task, so the GC could reap an in-flight `_bridge` mid-await → the warning (and a prematurely dropped worker connection). Now a closure-level `_bridge_tasks` set holds a strong ref to each until it finishes (`add_done_callback(discard)`). Validated: py_compile.

## server.py (0.2-m4c85)

m4c85 **Shard compiles run in a SUBPROCESS (GIL isolation) — fix "data connection closed" during compile.** Symptom: with two parallel int8 compiles running, live generation got bursts of `ConnectionError('data connection closed')` (one data-link drop dooms all queued requests at once); isolated requests still worked. Root cause: the compiles ran as in-process threads (`asyncio.to_thread`) and their Python-level work (the per-tensor loop + sha256) holds the GIL, starving the controller's SINGLE event-loop thread — even on the 124-core beast (only ~26 cores busy, so NOT CPU-bound; the GIL serializes Python to one thread). The stalled event loop drops the head→controller logits connection. Fix: `/compile_shards` now spawns each compile as a SEPARATE PROCESS (`asyncio.create_subprocess_exec`, its own GIL) running `shards.compile_shards`, and reads `P done total` / `DONE {json}` progress lines over the pipe asynchronously — the heavy work can't touch the controller's GIL/event loop, so parallel compiles no longer disrupt serving. Below-normal priority on Windows (`BELOW_NORMAL_PRIORITY_CLASS`) / `nice(10)` on Unix as a bonus. Falls back to the in-process path if the event loop can't spawn (Selector loop), logged.
