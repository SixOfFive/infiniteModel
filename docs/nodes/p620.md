# Node: `work` — Quadro P620 (Pascal, sm_61)

A small **GPU helper worker** in the InfiniteModel fleet. ~4 GB of VRAM means it contributes
only a handful of layers to a distributed (pipeline) split — it is a minor assist, not a
primary compute node. Dense int4 decode still runs fast on it (torch tinygemm); the fused-MoE
acceleration tier does **not** apply (Pascal is pre-Ampere — see [Gotchas](#6-gotchas)).

See also: [../ACCELERATION.md](../ACCELERATION.md) (int4 kernel matrix, the MoE opt-in), and
[../ROCM.md](../ROCM.md) (AMD nodes — not this box, but the Linux launch pattern is shared).

## 2. Install

NVIDIA on Linux → standard CUDA torch wheel + the normal app deps. No Triton toolchain is
needed here (the only thing Triton would buy on NVIDIA is the **fused-MoE opt-in**, which this
Pascal card cannot use — see Gotchas).

```bash
# 1) Python venv + build tools.
python3 -m venv ~/imenv

# 2) PyTorch (CUDA build). Use the CUDA index that matches the box's installed driver/toolkit.
#    (verify) the exact CUDA version against this host's `nvidia-smi`; cu128 shown as the
#    fleet's current build.
~/imenv/bin/python -m pip install torch --index-url https://download.pytorch.org/whl/cu128

# 3) App deps (same pinned versions as the rest of the fleet — from ROCM.md TL;DR):
~/imenv/bin/python -m pip install \
    transformers==5.12.1 safetensors==0.8.0 huggingface_hub==1.19.0 \
    numpy==2.4.6 psutil==7.2.2 einops fastapi uvicorn
```

> The `triton-windows` / MSVC / `ptxas` setup in [../ACCELERATION.md](../ACCELERATION.md)
> ("Windows + NVIDIA — enabling the advanced MoE tier") is **not relevant** to this node: it is
> Linux, and the MoE opt-in it enables won't run on Pascal anyway.

## 3. Run the worker

Join the existing fleet — launch **only** the worker, pointed at the fleet controller. Use
`--device cpu+gpu` so the box offers both its small GPU and CPU spill (matches the
[ROCM.md Launch](../ROCM.md#launch) worker pattern):

```bash
cd ~/infinitemodel
setsid ~/imenv/bin/python client.py --controller <controller-ip> --device cpu+gpu \
    >~/worker.log 2>&1 </dev/null &
```

- `--controller <controller-ip>` overrides `config.json`'s `controller_host`. **Don't** edit
  `config.json` — it's in `EXTRA_UPDATE_FILES`, so a self-update would revert your edit
  (per ROCM.md).
- Run **exactly one** worker per box (two workers sharing a hostname fight over the controller
  registration — ACCELERATION.md).
- **Do NOT** set `INFINITEMODEL_CUDA_FUSED_MOE` here (see Gotchas).

**Persistence:** for survival across reboot, run the worker under a `systemctl --user` unit and
enable lingering once so user services start at boot (the Linux method used on the fleet's
ROCm/Linux nodes — ROCM.md):

```bash
loginctl enable-linger "$USER"   # one-time
```

(verify) the exact unit file / service name in use on `work`; the standard fleet pattern is a
`--user` unit whose `ExecStart` is the `client.py` command above.

## 4. Optimal settings

Tuned for a **~4 GB** GPU tier:

- **Quant: int4.** Dense int4 decode uses torch **tinygemm** (`_weight_int4pack_mm`) on NVIDIA,
  which is the fast batch-1 path (and beats bf16) — ACCELERATION.md. int4 also quarters the
  bytes read per token, which matters most on a tiny card. Prefer int4 here.
- **bf16:** only viable for something that genuinely fits ~4 GB after KV — almost nothing useful
  does. Treat bf16 as not applicable on this node.
- **Context:** keep it **modest** — KV cache eats the scarce VRAM. Long contexts should land on
  bigger nodes; this box holds only a few layers anyway. Prefill chunking
  (`INFINITEMODEL_PREFILL_CHUNK`, default 2048) is on by default and is math-identical; you
  generally don't need to touch it on NVIDIA (the OOM it guards against is a ROCm/CPU-math-backend
  issue — ACCELERATION.md).
- **Placement mode:** this is a **distributed pipeline contributor** — it takes a small handful
  of a model's layers in a multi-node split. It is **not** a "load a whole model here" box.
  Prefer sending it a few layers of a model whose bulk lives on the larger GPU nodes.
- **What fits / runs well:** a few layers of any fleet model in a pipeline split. **Sparse MoE
  with a low active-param ratio** decodes fastest per the
  [ACCELERATION.md model-choice guidance](../ACCELERATION.md#choosing-a-fast-model-on-bandwidth-limited-gpus),
  but note the MoE *expert* fusion speedup does **not** apply here — only the dense tinygemm path.
- **What to avoid:** loading a full model onto this card; large contexts; and **the fused-MoE
  opt-in** (next section).

## 6. Gotchas

- **Pascal is pre-Ampere — do NOT set `INFINITEMODEL_CUDA_FUSED_MOE` on this box.** The fused
  MoE-expert kernel is **bf16 Triton**, which **cannot compile on sm_61**:
  `ptxas: Feature '.bf16' requires .target sm_80 or higher`. The load-time self-check catches the
  compile error and **safely falls back to the default** bf16-rematerialize path — so it's
  *harmless but ineffective* (no fused-MoE speedup, no breakage). Explicitly validated on the
  P620 in [../ACCELERATION.md](../ACCELERATION.md):
  *"Quadro P620 (Pascal sm_61) — Triton bf16 won't compile → safe automatic fallback (no
  speedup, no breakage)."* Leaving it unset is the correct configuration.

- **Tiny VRAM = tiny role.** ~4 GB only holds a handful of layers. Don't expect to host a model
  here; plan placements so the bulk sits on BEAST/larger GPUs and `work` carries a small slice.

- **Dense int4 still helps.** The lack of MoE fusion does **not** mean "no acceleration": the
  dense linears (q/k/v/o, shared expert) still go through the fast torch tinygemm int4 GEMM on
  this NVIDIA card. That path is automatic — nothing to enable.

- **One worker per host; don't edit `config.json`.** Two workers on the same hostname collide on
  registration; `--controller` is the right override (config.json is reverted by self-update).

- **Perf-only kernel changes don't auto-restart.** A self-update *stages* perf-only code (no
  VERSION bump) but won't restart the worker — relaunch it to pick up a kernel change
  (ACCELERATION.md). N/A for the MoE kernel here, but applies to any dense-path tweak.

- For anything not covered above (CUDA-graph decode, prefill-chunk tuning, the full kernel
  matrix), see [../ACCELERATION.md](../ACCELERATION.md); for AMD-node specifics see
  [../ROCM.md](../ROCM.md).
