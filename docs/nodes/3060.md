# Node: RTX 3060 (Ampere, sm_86)

Two RTX 3060s run in the fleet. They share an arch (**Ampere, compute capability sm_86**)
but differ in VRAM, OS, and persistence. Ampere is **sm_80+**, so the fused-MoE opt-in
(`INFINITEMODEL_CUDA_FUSED_MOE`) is supported on the Linux box — see
[../ACCELERATION.md](../ACCELERATION.md).

| Host | Card | VRAM | OS | Role | Persistence |
|---|---|---|---|---|---|
| **theocomp** | RTX 3060 | 12 GB | Linux | GPU worker (full layers) | systemd **system** unit, passwordless sudo |
| **mobile** | RTX 3060 Laptop | 6 GB | Windows | small/edge worker (few layers) | `start_worker.bat` / `client.bat` (interactive session) |

---

## 1. Overview

- **GPU arch:** Ampere, **sm_86** — important because it is **≥ sm_80**, the cutoff for the
  bf16 fused-MoE Triton kernel. The opt-in compiles and runs here (validated on the RTX 3060,
  rel 0.006, ~3.9× on the routed-expert GEMM).
- **theocomp (12 GB, Linux):** the workhorse 3060 — takes a real pipeline stage (many layers),
  and is the box where the advanced MoE tier is enabled.
- **mobile (6 GB, Windows):** the only 6 GB card in the fleet. Too small for a whole model of
  any size — it contributes a **few layers in a distributed split**, or hosts a small model
  outright. Windows + consumer GeForce, so it must run in the interactive desktop session
  (see Gotchas).
- Both serve int4 **dense** linears via torch **tinygemm** (`_weight_int4pack_mm`), the fast
  batch-1 path on NVIDIA — int4 beats bf16 on these cards.

## 2. Install

Both boxes need Python + the **CUDA 12.8 torch build** + the app deps. The 3060 is `cu128`-fine.

### theocomp (Linux)

```bash
# Python venv (3.14 in the fleet). Triton (for the fused-MoE opt-in) JIT-compiles launcher
# stubs, so it needs a C compiler + Python headers:
sudo apt-get install -y python3-venv python3-dev gcc git
python3 -m venv ~/imenv

# torch — CUDA 12.8 build:
~/imenv/bin/python -m pip install torch --index-url https://download.pytorch.org/whl/cu128

# App deps (same versions as the rest of the CUDA fleet — from ROCM.md TL;DR):
~/imenv/bin/python -m pip install \
    transformers==5.12.1 safetensors==0.8.0 huggingface_hub==1.19.0 \
    numpy==2.4.6 psutil==7.2.2 einops fastapi uvicorn

# Sanity check — must print True and run a matmul with no crash:
~/imenv/bin/python -c "import torch; \
print('cuda', torch.version.cuda, '| avail', torch.cuda.is_available(), \
'|', torch.cuda.get_device_name(0)); \
a=torch.randn(2048,2048,device='cuda'); (a@a).sum().item(); print('matmul OK')"
```

> Triton ships with the Linux torch wheel; `gcc` + `python3-dev` are what its JIT needs. The
> sanity-check / dep-version lines are adapted from [../ROCM.md](../ROCM.md) (same fleet deps,
> CUDA index substituted for ROCm).

### mobile (Windows)

torch **2.11+cu128** is installed under **`C:\Python314`** (user-site, CPython 3.14). The worker
runs from **`c:\infinitemodel`** via **`client.bat`**.

```bat
:: From a normal cmd (NOT a service) on mobile:
C:\Python314\python.exe -m pip install --user torch==2.11.* --index-url https://download.pytorch.org/whl/cu128
C:\Python314\python.exe -m pip install --user transformers==5.12.1 safetensors==0.8.0 huggingface_hub==1.19.0 numpy==2.4.6 psutil==7.2.2 einops fastapi uvicorn
```

> ⚠ Reinstalling torch in place on mobile leaves a broken mixed tree — **stop the worker first**
> (it locks the DLLs), wipe `site-packages/torch`, then clean-install. (verify exact wipe path
> against this box's user-site.)
> The fused-MoE opt-in is **not** used on mobile (Windows + only 6 GB), so the Triton-on-Windows
> toolchain (Build Tools + CUDA Toolkit `ptxas`) from [../ACCELERATION.md](../ACCELERATION.md) is
> **not required** here.

## 3. Run the worker

### theocomp — systemd **system** unit

Launch (the line the unit runs; `--device gpu` for a pure-GPU stage, or `cpu+gpu` to allow CPU
spill — match the unit). Point `--controller` at the fleet controller (BEAST), not localhost:

```bash
~/imenv/bin/python client.py --controller <controller-ip> --device gpu
```

Enable the **advanced MoE tier** via a systemd **drop-in** (this is how it's set on theocomp):

```ini
# /etc/systemd/system/infinitemodel-worker.service.d/fused-moe.conf
[Service]
Environment=INFINITEMODEL_CUDA_FUSED_MOE=1
```

```bash
sudo systemctl daemon-reload
sudo systemctl restart infinitemodel-worker
```

The fused forward installs at **load time**, so restart the worker and reload the model after
changing the env. (verify the exact unit name / ExecStart on theocomp.)

### mobile — `client.bat` (interactive session)

`client.bat` cd's to `c:\infinitemodel` and launches `client.py` against the controller. The
device flag is `--device gpu`. Persistence is the batch file via `start_worker.bat` /
`shell:startup` — **not** a service (see Gotchas: WDDM/session-0).

```bat
:: client.bat (mobile) — shape, adapt the controller IP to the fleet:
cd /d c:\infinitemodel
C:\Python314\python.exe client.py --controller <controller-ip> --device gpu
```

> For exact flags not shown (`--control-port`, `--data-port`, etc.) and the standalone
> controller+worker pattern, see the Launch section of [../ROCM.md](../ROCM.md) — the
> `client.py` interface is identical on CUDA.

## 4. Optimal settings

**Quant:** **int4** on both. Dense int4 uses torch tinygemm on Ampere — already the fast batch-1
path, and int4 beats bf16 (int4 7B 3.25 → 21.13 tok/s in [../ACCELERATION.md](../ACCELERATION.md)).
Use bf16 only if a small model fits comfortably and you want to skip dequant.

### theocomp (12 GB)

- **Models that fit/run well:** a ~7B dense int4 fits whole; a small **MoE** (e.g. `olmoe:1b-7b`)
  is ideal — low active-param ratio → fast decode, and the fused-MoE opt-in gives the routed
  experts ~3.7× (validated: `olmoe:1b-7b` int4 **8.5 → ~31.5 tok/s** with `INFINITEMODEL_CUDA_FUSED_MOE=1`).
- **Placement:** as a real GPU stage in a pipeline split, or all-GPU for a model that fits in 12 GB.
- **Context:** moderate — KV competes with weights in 12 GB. Long prefill is chunked by default
  (`INFINITEMODEL_PREFILL_CHUNK=2048`); lower it for very long contexts on a tight box.
- **Prefer:** sparse MoE with low active params over a big dense model of the same footprint
  (decode tracks active bytes/token).

### mobile (6 GB)

- **Best use:** a **few layers in a distributed split**, or a small model (≤ a few B) at int4.
- **Avoid:** loading a full mid/large model here — 6 GB is the smallest tier in the fleet;
  CPU spill on a model that doesn't fit causes contention.
- **Placement:** contributory stage in a pipeline, int4, modest context.

> On bandwidth-limited cards, prefer a sparse **MoE (~3B active)** over a dense model of similar
> quality — see "Choosing a fast model" in [../ACCELERATION.md](../ACCELERATION.md).

## 5. Verify

**Client self-test (load on GPU)** — confirms the worker can load weights onto the 3060:

```bat
:: mobile
client.bat --self-test-load --device gpu --quant int4
```

```bash
# theocomp (equivalent via client.py)
~/imenv/bin/python client.py --self-test-load --device gpu --quant int4
```

**Registration check** — the worker should appear in the controller's status with its GPU + VRAM:

```bash
curl -s http://<controller-ip>:21434/status
```

**Fused-MoE engaged (theocomp only)** — load a MoE and look for the self-check in the worker log:

```bash
curl -s "http://<controller-ip>:21434/logs?node=theocomp"
# expect:  [int4] fused-MoE self-check ... -> ACTIVE
#  (-> fallback means it self-checked out / toolchain incomplete → safe bf16-remat path)
```

**Gen test** — a quick coherent-generation through the controller API confirms end-to-end
(load a model, then a short completion; "Paris" smoke test as used across the fleet).

## 6. Gotchas

- **`INFINITEMODEL_CUDA_FUSED_MOE=1` is Linux + Ampere only.** It engages on theocomp (sm_86 ≥ sm_80).
  It is **opt-in**, not default — Triton is unreliable to build on Windows, so most NVIDIA boxes
  leave it off. On a pre-Ampere card the bf16 kernel won't compile and the self-check falls back
  safely. The opt-in upgrades only the **routed-expert** GEMV; dense already uses tinygemm. See
  [../ACCELERATION.md](../ACCELERATION.md).
- **mobile must run in the interactive desktop session.** A consumer GeForce under the WDDM driver
  is **not visible to a Windows service or a "run whether logged on or not" scheduled task**
  (session 0). Launch the worker from `cmd` / a shortcut / `shell:startup` — never a service or
  Task Scheduler "run whether logged on or not" entry. (From the WDDM gotcha in [../ACCELERATION.md](../ACCELERATION.md).)
- **mobile torch reinstall is fragile.** Stop the worker (it locks the DLLs) before touching
  `site-packages/torch`; an in-place reinstall leaves a broken mixed tree — wipe then clean-install.
- **One worker per box.** Two workers sharing a hostname fight over the controller registration.
- **Perf-only kernel changes carry no VERSION bump** — a self-update *stages* the new code but does
  **not** auto-restart the worker. Relaunch (`client.bat` / `systemctl restart`) to pick up a kernel
  change. (From [../ACCELERATION.md](../ACCELERATION.md).)
- **Env changes apply at load time.** After setting/clearing `INFINITEMODEL_CUDA_FUSED_MOE` (or any
  switch), restart the worker **and reload the model** — the fused forward installs during load.
- **Don't edit `config.json` to point the worker** — it's in `EXTRA_UPDATE_FILES` and a self-update
  would revert it. Use `--controller` on the launch line. (From [../ROCM.md](../ROCM.md).)
- **Long-prompt prefill** is chunked by default (`INFINITEMODEL_PREFILL_CHUNK=2048`, standard-attention
  only). On a memory-tight box (esp. 6 GB mobile) lower it (512–1024) for very long contexts; `0`
  disables. See [../ACCELERATION.md](../ACCELERATION.md).
